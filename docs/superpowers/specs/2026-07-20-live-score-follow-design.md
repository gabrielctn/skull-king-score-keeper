# Live score follow via QR code — design

Date: 2026-07-20
Status: implemented

## Goal

The game master keeps the score on one device; everyone else has to ask
"did you record my bonus?". Give every player a way to check their own
detailed score — bids, tricks, every bonus line — on their own phone,
without touching the game master's device and without adding a server to
an offline-first PWA.

## Decisions

### The QR code is the transport, not a pointer

There is no backend, and games are played at tables with no wifi, so a
"live session" service is out. Instead the **whole game snapshot travels
inside the QR code**: a `#skl=<code>` hash on the app's own URL. Scanning
opens the deployed PWA (from cache when the player has already visited or
installed it) and the hash carries all the data — the two devices never
talk to each other.

Consequences embraced:

- The player's view is a **snapshot**; re-scanning refreshes it. The
  spectator screen says so and shows the read-out time. This matches the
  actual need ("check that my bonus was recorded"), which is occasional.
- The QR shown by the game master always encodes the **latest saved
  state** — the sheet re-renders on every persisted change, so there is
  nothing to sync or forget.

### Compact binary codec instead of JSON in the URL

A full game as JSON is tens of kilobytes — beyond QR capacity. The
`src/shareLink.ts` codec writes varints, bit-flag bytes and UTF-8 names,
then base64url-encodes; empty bonus blocks cost one flag bit. Measured
sizes: a fresh 2-player game ≈ 155 chars; the theoretical worst case
(8 players × 10 rounds, every bonus field set, loot everywhere) ≈ 1 560
chars — a scannable QR at error-correction level M, generated by the
zero-dependency `qrcode-generator` package.

Player IDs are not transmitted: seats are, and the decoder derives
`shared_p<seat>` IDs. Loot pairings reference seats.

### Untrusted input goes through the backup hardening

A share code is attacker-controllable input. `decodeShareCode` bounds
every count while parsing, then runs the result through
`normalizeUntrustedGame` — the same validation + migration + deep-copy
path used for imported backup files (`backup.ts`). Anything off throws a
typed `ShareCodeError`; the spectator screen then shows a friendly
"code could not be read" view instead of crashing.

### Spectator mode is a mode, not a saved game

`App.tsx` resolves the hash **synchronously in a lazy `useState`
initializer**, strips it from the address bar immediately (so the payload
— it contains player names — never reaches analytics, history or
bookmarks), and stashes the code in `sessionStorage` so a reload stays in
spectator mode. A `hashchange` listener handles re-scans while the app is
open. Exiting clears the session code and reveals the normal app.

The decoded game is **never merged** into the device's current game or
history; the only thing persisted is a small "I am player X in game Y"
record (`skullking:spectatorIdentity`), so the next scan of the same game
reopens that player's score details automatically (name-checked, in case
seats ever disagree).

### Reused UI instead of a new score view

The spectator screen shows standings (rank, name, total, "You" chip) and
opens the existing `ScoreBreakdownModal` — the same per-round detail the
game master sees, bonuses and ignored-item annotations included — plus
the existing `ScoreChart` once two rounds are scored. The game master's
entry point is a ▦ button next to the rules "?" in the game header,
opening `ShareLiveModal` (QR on a white card, scan/refresh/offline hints,
copy-link fallback for sharing through a messenger).

## Testing

`scripts/test-sharelink.ts` (in `npm test`): round-trip fidelity down to
each bonus field and loot seat mapping, standings equality between the
master's game and the decoded snapshot, deterministic encoding, size
bounds, QR generation, URL helpers, and tamper rejection (truncated,
mutated, oversized, wrong-version and garbage codes must throw
`ShareCodeError`, never crash). The full master → scan → check-bonus →
reload → re-scan → exit flow is driven end-to-end with Playwright.
